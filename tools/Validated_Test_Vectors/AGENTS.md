# AGENTS.md

This file provides guidance to coding assistant agents when working with code in this directory. It complements the top-level `CLAUDE.md` at the repo root.

## Purpose

A standalone Lean 4 / Mathlib project that formally proves correctness properties of the APRS Base91 compressed-position encoding implemented (in C++) by the rest of this repository, **and** acts as the on-disk oracle for the lat/lon bytes emitted by the sibling Python generator.

- Proofs: `encodeLat_decodeLat_roundtrip`, `encodeLon_decodeLon_roundtrip` in `ValidatedTestVectors/Basic.lean`.
- Generator: `Main.lean` writes `proved_vectors.json` containing the formally-proven `encodeLat`/`encodeLon` outputs for the same fixed-point sites that `tools/gen_aprs_vectors.py` uses (Munich, Cape Town, Auckland, North Pole, Equator-0).
- Consumer: `tools/gen_aprs_vectors.py` reads `proved_vectors.json` and threads its `base91_lat`/`base91_lon` fields directly into `test/common/aprs_vectors.h`. The Python script's `encode_lat_base91`/`encode_lon_base91` functions remain as a reference implementation only — they are not on the production path.

The fixed-point list lives in **both** `Main.lean` (`fixedPoints`) and `gen_aprs_vectors.py` (`FIXED_POINTS`). They must match. The Python loader cross-checks names and lat/lon to first order before using any byte, and aborts loudly if they disagree.

## Layout

- `lakefile.toml` — Lake build manifest. Pins `mathlib` to `v4.28.0`. The `[leanOptions]` block carries the project-wide elaborator and pretty-printer knobs (heartbeats, `autoImplicit false`, `pp.*`, etc.); these apply to every file in the package so individual `.lean` files do not need their own `set_option` headers.
- `lean-toolchain` — pins the Lean compiler to `leanprover/lean4:v4.28.0`. **Must match the Mathlib rev.** When bumping Mathlib, bump both files together.
- `Main.lean` — entry point of the `validated_test_vectors` executable, and the test-vector generator. Its `main` writes `proved_vectors.json` to the current working directory using the formally-proven `encodeLat`/`encodeLon` from `Basic.lean`.
- `proved_vectors.json` — generated. Committed. Consumed by `tools/gen_aprs_vectors.py`. Regenerate after touching `fixedPoints` in `Main.lean`.
- `ValidatedTestVectors.lean` — library root, just re-exports `ValidatedTestVectors.Basic`.
- `ValidatedTestVectors/Basic.lean` — the actual content: Base91 encode/decode, the round-trip theorems (`encodeLat_decodeLat_roundtrip`, `encodeLon_decodeLon_roundtrip`), and sample `example`s at concrete inputs.
- `lake-manifest.json` — generated, but committed. Regenerated by `lake update`. Do not hand-edit.
- `.lake/` — build cache, gitignored.
- `.github/workflows/lean_action_ci.yml` — runs `leanprover/lean-action@v1` on push/PR. This is the only CI in the wider repo, and it only covers this subdirectory.

## Build / verify / generate

From this directory:

```sh
lake build ValidatedTestVectors    # type-checks the library — this is the proof suite
lake build                         # also tries to link the exe (see gotcha below)
lake env lean --run Main.lean      # generates proved_vectors.json (works on macOS)
lake exe validated_test_vectors    # same, via the linked exe (Linux/CI only on macOS)
```

First build downloads Mathlib and can take 10+ minutes (or a few minutes if the Mathlib build cache is reachable via `lake exe cache get`). Subsequent builds are incremental. **Type-checking is the test suite here** — if `lake build ValidatedTestVectors` succeeds, the proofs are valid; there is no separate `lake test` step.

Run `lake build ValidatedTestVectors` (not just a syntax check) before claiming a proof goes through. Lean tactic proofs frequently parse fine but fail to elaborate.

### macOS exe-link gotcha

On macOS, plain `lake build` (which builds the `validated_test_vectors` exe target named in `defaultTargets`) fails at the final link step with:

```
could not execute external process '…/clang'
error: external command '…/clang' exited with code 255
```

The cause is *not* a broken clang — Lake invokes it with ~8,000 `.o.export` object-file arguments on a single command line, exceeding macOS's per-process argv+env limit, and the kernel rejects the `execve`. The library compiles and all proofs elaborate (you'll see `Built ValidatedTestVectors.Basic` and the `#eval` outputs before the failure); only the exe link is broken.

Workarounds, in order of pragmatism:

- **For type-checking proofs only**: `lake build ValidatedTestVectors`. Skips the exe target entirely.
- **For regenerating `proved_vectors.json`**: `lake env lean --run Main.lean`. This invokes the Lean interpreter on the source, which uses the already-built `.olean` files and skips the C codegen + link path the exe target uses. Verified to work on macOS.
- **Trust CI** for the exe link itself. The GitHub workflow runs on Linux, where `ARG_MAX` is large enough.
- A real fix is upstream-flavoured (response-file linking, slimmer transitive deps, or `--whole-archive`-style packing) — don't paper over it with `--no-verify`-style escapes.

When reporting a Lean change as "verified", say which command you ran. `lake build ValidatedTestVectors` succeeding is sufficient evidence for proof correctness; the exe link failing on macOS is *not* a regression you introduced.

## What is actually proved

Two headline theorems in `Basic.lean`:

```lean
theorem encodeLat_decodeLat_roundtrip (x : ℚ) (hlo : -90 ≤ x) (hhi : x ≤ 90) :
    BoundedRoundTrip encodeLat decodeLat aprs_epsilon x

theorem encodeLon_decodeLon_roundtrip (x : ℚ) (hlo : -180 ≤ x) (hhi : x ≤ 180) :
    BoundedRoundTrip encodeLon decodeLon lonEpsilon x
```

`BoundedRoundTrip enc dec ε x` states `|dec (enc x) - x| ≤ ε`. The bounds are:

- `aprs_epsilon = 1/300000` — safe upper bound on the latitude round-trip error (the true bound is `1/latFactor = 1/380926`; the looser constant gives margin vs. the `1/12000` ties-to-even APRS LSB).
- `lonEpsilon = 1/150000` — same idea on the longitude side, where `lonFactor = 190463` (half of `latFactor`), so the worst-case error is roughly doubled.

The proof chain (latitude; longitude mirrors it):

1. `charToVal_valToChar` — single-character round-trip, proved by `decide +revert`.
2. `foldl_toBase91_loop` — the generalised induction hypothesis. Load-bearing lemma; parameterised over `init` and `acc` so the induction goes through.
3. `fromBase91_toBase91` — string-level round-trip, given `n < 91^len`. Shared by both latitude and longitude branches.
4. `encodeLat_in_range` / `encodeLon_in_range` — the floored-and-scaled coordinate is `< 91^4`, so `fromBase91_toBase91` applies. Longitude's bound is tight: `360 * 190463 = 68566680 < 91^4 = 68574961`.
5. `encodeLat_decodeLat_roundtrip` / `encodeLon_decodeLon_roundtrip` — combine (3) and (4) with `floor_error_nonneg` / `floor_error_lt_one` to bound the residual.

The ten `example`s near the bottom of `Basic.lean` (five per coordinate) are *not* unit tests in the runtime sense — they are additional Lean obligations that re-check each theorem at concrete inputs at elaboration time. They cost build time but document boundary values (`±90`/`±180`, `0`, off-axis).

## Known limitations (don't mistake these for bugs)

- **`toBase91_loop_eq_append` is commented out** (lines 103–110-ish). It was an intermediate lemma that the current proof of `foldl_toBase91_loop` does not need. Leave it commented unless a future proof requires the append form — don't delete the comment block, the structure is useful as a reference.
- The proofs use `ℚ` (rationals), not `ℝ`. The C++ implementation uses floats. The bound therefore certifies the *mathematical* formula, not the floating-point implementation; rounding error in the C++ float arithmetic is a separate concern and is *not* covered here.
- `Main.lean`'s `fixedPoints` list duplicates `FIXED_POINTS` in `tools/gen_aprs_vectors.py`. Keep them in lock-step. The Python consumer validates the name + lat/lon match before using any byte, and aborts loudly otherwise.

## Style observations

- Heavy use of one-liner Lean tactic combos (`simp_all +decide [ ... ] ;` chains, `linarith!`, `exact_mod_cast`). When extending, match the existing terse style — multi-line structured proofs are not the convention here.
- Spaces inside brackets (`[ Nat.mod_one ]`, `( by decide )`) are an existing stylistic choice; don't reformat to canonical Mathlib style.
- `autoImplicit false` is set project-wide in `lakefile.toml`. New definitions must declare their implicit binders explicitly.
- The five `open scoped` declarations in `Main.lean` (`BigOperators`, `Real`, `Nat`, `Classical`, `Pointwise`) are not all needed by the current code — they're present as a permissive default for proofs that may live in `Main` later.

## When extending

- Add new theorems to `ValidatedTestVectors/Basic.lean` (or a new file imported by `ValidatedTestVectors.lean`). Don't add proof content to `Main.lean`.
- After any change, run `lake build` and only report success if it exits clean. Lean errors are *not* warnings — partial proofs leave the file unusable downstream.
- When bumping Mathlib, update both `lakefile.toml` (`rev = "..."`) and `lean-toolchain`, then run `lake update && lake build` and commit the regenerated `lake-manifest.json`.
